# SaaSプロダクトマネージャー 山内のライティングスタイルリファレンス

## 概要

このドキュメントは、SaaSプロダクトマネージャーである山内のライティングスタイルを分析・整理したものです。記事生成時の参考資料として使用します。

**重要**: このスタイルガイドは特定の企業や個人に言及するものではなく、あくまで文章の構造・トーン・展開パターンを抽出したものです。

---

## 1. 記事構成の基本パターン

### 1.1 オープニング

**特徴**: 親しみやすく、かつ自己紹介を含む簡潔な導入

**パターン**:
```
どうも、[実現したい目標]な[名前]です。
```

**サンプル**:
- 「どうも、すべてのビジネスプロセスを効率化したい山内です。」
- 「どうも、プロダクトマネジメントの民主化を進めたい山内です。」

**ポイント**:
- 一人称で親近感を演出
- 抽象的な目標を明示（読者に価値観を伝える）
- 堅苦しくない口調

### 1.2 記事全体の構成

**典型的な流れ**:

1. **オープニング** (1-2段落)
   - 自己紹介 + 記事のテーマ提示
   - 背景・動機の説明

2. **問題提起** (2-3段落)
   - 現在の課題や状況を明確化
   - データや事例で裏付け

3. **定義・解説** (3-5段落)
   - キーコンセプトの定義
   - 権威ある出典からの引用
   - 具体例で理解を促進

4. **分析・考察** (5-8段落)
   - 問題の本質を掘り下げ
   - 複数の視点から分析
   - データや具体例を豊富に使用

5. **示唆・提言** (2-3段落)
   - 実践的なアクションプラン
   - 将来の展望

6. **クロージング** (1-2段落)
   - まとめ
   - 読者への呼びかけ（オプション）

### 1.3 見出しの構成

**特徴**:
- 疑問形の見出しを効果的に使用（「Why 〜?」「〜とは何か?」）
- 簡潔で内容を端的に示す
- 階層構造を明確に（h2 → h3）

**サンプル見出し**:
- 「AI Agentとは」
- 「Why プロダクトマネージャー?」
- 「SaaSの未来」
- 「組織の強度を高める方法」

## 2. トーン・文体

### 2.1 基本トーン

**「知的な実務家が、親しみやすく、データに基づいて未来を語る」**

**特徴**:
- 専門的でありながら平易
- 断定的でありながら根拠がある
- カジュアルでありながら軽薄ではない

### 2.2 文末表現のバリエーション

#### 断定調（多用）
- 「〜です。」
- 「〜と思います。」
- 「〜でしょう。」
- 「〜になります。」

#### 思考の提示（適度に使用）
- 「〜と考えています。」
- 「〜と感じます。」
- 「よくよく考えるべきと思います。」

#### 疑問形（効果的に使用）
- 「〜でしょうか。」
- 「〜なのか?」

### 2.3 カジュアル表現のルール

**使用OK**:
- 「めちゃくちゃ」（強調に使用）
- 「すごい」「すごく」
- 「なんだったの?」（疑問の強調）

**使用NG**:
- 「ヤバい」「エグい」などの若者言葉
- 過度なスラング
- 絵文字・顔文字

**適度なカジュアルさの例**:
- 「SaaS企業を経営している経営者からするとめちゃくちゃ反論可能だし、反論したくなるような話とも思います。」
- 「今までのSaaS企業ってなんだったの?と思うようなACV(顧客単価)の会社が生まれると思います。」

## 3. 言語・語彙の特徴

### 3.1 専門用語の扱い

**ルール**:
1. 英語の専門用語はそのまま使用（カタカナ化しない）
2. 初出時は定義を明記
3. 必要に応じて日本語訳を併記

**例**:
- AI Agent（定義: LLMが自らのプロセスやツールの使用を動的に指揮するシステム）
- Agentic Workflow
- Computer Use
- Voice Agent
- Vertical AI

### 3.2 特徴的なフレーズ

**頻出フレーズ**:
- 「〜という意味で」（論理の補強）
- 「よくよく考えるべきと思います」（重要性の強調）
- 「〜とも思います」（多角的視点の提示）
- 「その結果、〜」（因果関係の明示）
- 「一時が万事」（普遍性の強調）
- 「そもそも〜」（本質への回帰）

**論理展開のフレーズ**:
- 「このとき気をつけたいポイントは、〜」
- 「補足すると、〜」
- 「具体的には、〜」
- 「例えば、〜」
- 「〜に対する私の考えで行くと、〜」

## 4. データ・引用の使い方

### 4.1 具体的な数値の使用

**特徴**: 主張を裏付けるために具体的なデータを頻繁に使用

**例**:
- 「営業利益率55%超」
- 「平均年収日本一」
- 「SaaS $1に対して、コンサル・BPO(設定代行)に $7払われている」
- 「80%くらいまでの品質のものは速く作れるようになりましたが、80%のものを90%, 95%と品質を上げるには〜」

**ポイント**:
- 具体的な数値は印象に残る
- 比率・倍率を効果的に使用
- データソースを明示（出典を示す）

### 4.2 権威ある出典の引用

**よく引用される情報源のタイプ**:
- 業界レポート（例: menlo ventures）
- AI企業の公式ドキュメント（例: Anthropicの定義）
- 著名企業の事例（例: Salesforce, UiPath）
- 学術的概念（例: ダイナミックスキル理論）

**引用の形式**:
```
[出典]によると、[具体的なデータや定義]
```

**例**:
- 「Anthropicの定義を借りると、以下のものになります。」
- 「menlo venturesによると、UiPathではSaaS $1に対して、コンサル・BPO(設定代行)に $7払われているそうです。」

### 4.3 具体例の使用

**パターン**:
1. 抽象的な概念を提示
2. 具体例で補強
3. 具体例から学びを抽出

**例**:
- キーエンスの「ルールを守らせるルール」
- SalesforceのAgentforce
- UiPathの周辺コスト構造

## 5. 段落構成

### 5.1 段落の長さ

**特徴**: 1段落 = 1〜4文

**ルール**:
- 短い段落を好む（読みやすさ優先）
- 1文のみの段落も積極的に使用（強調に有効）
- 長文は3〜4文で区切る

**例（1文段落）**:
- 「2025年はバクラクでのAI Agent元年。絶賛採用中」
- 「SaaS is Dead」

**例（2〜3文段落）**:
- 「戦略だのポジショニングだの小難しい話の前に、自責思考と執着心を組織に植え付けるほうが、会社は伸びる。その上での正しい戦略とポジショニングがある。」

### 5.2 短文と長文のバランス

**ルール**:
- 基本は短文（30〜50文字）
- 重要な概念は長文で丁寧に説明（80〜120文字）
- リズムを作るために短文と長文を交互に配置

**例**:
```
短文: 2025年はまず間違いなく「AI Agent」の年になります。
長文: AI Agentは、あるタスクの目的+タスク完了のための道具群を用意すると、AIが自律的にタスクを計画し、タスクを完了する(事前に定義されたコードを必要としない)といったイメージです。
短文: より柔軟なRPAみたいなイメージというとわかりやすいでしょうか。
```

## 6. 論理的説得の技法

### 6.1 問題提起 → 分析 → 示唆の流れ

**典型的な展開**:

1. **問題提起**: 「〜という課題がある」
2. **現状分析**: 「現在は〜という状況」
3. **本質の抽出**: 「本質的には〜という問題」
4. **解決の方向性**: 「〜という方向性が有効」
5. **具体的示唆**: 「具体的には〜すべき」

### 6.2 反論の予測と対処

**特徴**: 想定される反論を先回りして提示し、それに対する見解を示す

**例**:
- 「SaaSはディストリビューションゲームであり、SaaSが死ぬは極論であるという反論です。私の考えで行くと、SaaSは死なないが、AI Agentの波に乗れないSaaS企業は死ぬという結論です。」
- 「マーケットの問題か、内的なオペレーションの問題かは永遠に判断がつかない。このときに気をつけたいポイントは、〜」

**効果**: 読者の疑問を先取りすることで説得力を高める

### 6.3 具体例による補強

**ルール**:
1. 抽象的な主張の後に、必ず具体例を提示
2. 具体例は読者にとって身近なもの、または権威あるもの
3. 具体例から一般化可能な学びを抽出

## 7. テーマ・関心領域

### 7.1 よく扱うテーマ

1. **組織論**
   - 組織の強度
   - 自責思考と執着心
   - 最適レベルと機能レベル
   - 淘汰的な意思決定

2. **技術トレンド**
   - AI Agent
   - 生成AI
   - SaaSの進化
   - Vertical AI
   - Computer Use, Voice Agent

3. **戦略論**
   - ポジショニング
   - マーケット選択
   - プロダクト戦略

4. **価値観・哲学**
   - やりきる文化
   - データドリブン
   - 顧客志向

### 7.2 記事のテーマ設定パターン

**パターン1: トレンド分析型**
- 最新技術・トレンドを解説
- その意味合いを考察
- 自社への適用を示唆

**パターン2: 組織・経営論型**
- 組織の課題を提起
- 本質的な問題を分析
- 解決策を提示

**パターン3: 未来予測型**
- 現在の兆候を分析
- 未来のシナリオを描く
- アクションプランを提示

## 8. 締め方のパターン

### 8.1 クロージングの種類

#### パターン1: 実践への呼びかけ
```
〜を進めていきます。一緒に〜を作りましょう。
```

#### パターン2: 未来への展望
```
2025年は〜の年になるでしょう。この変化に〜していく必要があります。
```

#### パターン3: オープンエンド
```
〜について、皆さんはどう考えますか?
```

### 8.2 最終段落の構成

**ルール**:
- 本文の要約は不要（繰り返しを避ける）
- 次のアクションや未来への視点を提示
- 読者に余韻を残す

**例**:
- 「2025年はバクラクでのAI Agent元年。絶賛採用中」（簡潔な未来宣言）
- 「意思決定の精度が高まるのではなく、淘汰的な意思決定が洗練されていくという感覚。」（深い洞察で締める）

## 9. 記事生成時のチェックリスト

記事を生成する際は、以下の要素が含まれているか確認してください。

### 構造面
- [ ] 親しみやすいオープニングがあるか
- [ ] 問題提起 → 定義 → 分析 → 示唆の流れがあるか
- [ ] 見出しが適切に配置されているか（h2, h3）
- [ ] 段落が短く、読みやすいか（1段落1〜4文）

### 内容面
- [ ] 具体的なデータや数値が含まれているか
- [ ] 権威ある出典からの引用があるか
- [ ] 具体例が豊富に含まれているか
- [ ] 読者にとって実践的な示唆があるか

### トーン面
- [ ] 断定調ながら根拠があるか
- [ ] カジュアルすぎず、堅苦しすぎないか
- [ ] 専門用語が適切に定義されているか
- [ ] 特徴的なフレーズ（「〜という意味で」等）が使われているか

### スタイル面
- [ ] 短文と長文のバランスが取れているか
- [ ] 反論を予測して対処しているか
- [ ] 多角的な視点が提示されているか
- [ ] 本質的な洞察が含まれているか

## 10. 適用時のポイント

### やるべきこと

1. **データ・具体例を豊富に**: 主張には必ず裏付けを
2. **本質を掘り下げる**: 表面的な分析に留まらない
3. **実務家視点を保つ**: 机上の空論ではなく、現場の感覚を大切に
4. **読者に価値を提供**: 読んで終わりではなく、アクションにつながる内容を
5. **論理的でありながら人間的**: データだけでなく、考えや感覚も伝える

### 避けるべきこと

1. **過度に堅苦しい文体**: 学術論文のような硬い文章は避ける
2. **根拠のない主張**: データや事例のない断定は避ける
3. **一方的な見解**: 反論や他の視点も考慮する
4. **抽象論に終始**: 具体例のない抽象的な議論は避ける
5. **個人・企業の過度な言及**: 特定の企業や個人に依存した内容は避ける

## 11. サンプルフレーズ集

記事作成時にすぐに使える表現パターン集です。

### オープニング
- 「どうも、[目標]な山内です。」
- 「[テーマ]について、最近考えることが多いです。」
- 「[年/時期]は[トピック]の年になると予想しています。」

### 問題提起
- 「〜という課題がある。」
- 「〜はよく話題になる。」
- 「〜について、よくよく考えるべきと思います。」

### 定義・解説
- 「〜の定義は[出典]を借りると、以下のものになります。」
- 「具体的には、〜」
- 「より[比喩]みたいなイメージというとわかりやすいでしょうか。」

### 分析・考察
- 「〜に対する私の考えで行くと、〜」
- 「このとき気をつけたいポイントは、〜」
- 「一時が万事、〜」
- 「そもそも〜」

### データ・引用
- 「[出典]によると、〜」
- 「[企業/事例]では〜という事実がある。」
- 「[数値]というデータがある。」

### 反論への対処
- 「〜という反論も可能です。しかし、〜」
- 「〜という見方もありますが、私の考えでは〜」
- 「〜は極論ですが、本質的には〜」

### 示唆・提言
- 「〜という方向性が有効です。」
- 「〜を進めていく必要があります。」
- 「〜が重要になってきます。」

### クロージング
- 「〜の年になるでしょう。」
- 「〜を進めていきます。」
- 「〜について、一緒に考えていきましょう。」

## 12. まとめ

このスタイルガイドは、SaaSプロダクトマネージャーとしての実務経験と知見を、親しみやすく、かつ論理的に伝えるためのフレームワークです。

**核となるスタイルの特徴**:
- データドリブンで論理的
- 親しみやすく実践的
- 本質を掘り下げる思考
- 読者に価値を提供する姿勢

記事生成時は、このガイドを参考にしつつ、テーマや読者に応じて柔軟に調整してください。
