AI BPO - Ambient Agent(アンビエントエージェント) + Human in the Loopで実現するAIエージェントの新標準
AI BPOとは
AI BPOとは、AIエージェントが業務を自律的に実行し、人が確認・修正した最終アウトプットを返すサービス形態です。 LayerXでは、Ambient AgentとHuman in the Loopの組み合わせで、AI BPOを提供しています。

AI BPOは

エンドユーザーはAIを操作もマネージもしない

AIのみで完結させず、人も組み込む(Human in the Loop)

「気づいたら仕事が終わっている」体験を提供する

という特徴を持ちます。

Zero-touch automationとAmbient Agent
画像
Zero-touch automation & Ambient Agentとチャット基点のAgentの違い
AI BPOのように「気づいたら仕事が終わっている」体験を「Zero-touch automation(ゼロタッチオートメーション)」と呼びます。従来のソフトウェアは人が起動し操作します。Zero-touch automationでは、ソフトウェアが自律的に判断・実行し、人の介在なしに仕事を完了させます。

この際のエージェントのあり方を「Ambient Agent(アンビエントエージェント)」と呼びます。Ambient Agentとは、チャット起点ではなく、イベント(メール受信、ファイルアップロード、フォーム送信など)をトリガーに自律的に起動し、タスクを完了するAIエージェントのことです。 「気づいたら仕事が終わっている」体験を実現します。

ただし現時点の前提で、AIだけに自律的に仕事をさせると十分な精度を出すことは難しいです。そこでAmbient AgentとHuman in the Loopを組み合わせます。

Human in the Loop(HITL)とは、AIの出力を人が確認・修正するプロセスを組み込んだシステム設計のことです。AI BPOでは、このHITLを組み込み、AIエージェントが主体的にタスクを実行しつつ、必要なタイミングで人が介入します。

私は、ほとんどのAIエージェントはAmbient Agent + Human in the Loopの組み合わせで提供されるのが最適だと思ってます。Ambient AgentがSaaSにネイティブに組み込まれ、自律的に仕事を進めていく体験であるべきだと考えています。その対極にあるのが、すべてをチャット起点で依頼する体験(Chat Agent) です。これはユーザーにとって必ずしも良い体験とは言えないと思います。もちろんチャット起点が適している領域も存在しますが、その範囲は今想定されているよりずっと狭いはずです。

どこをAIに任すか、どこを人に任すかをユーザー側でなく、オペレーター側が設計することがAI BPOのポイントです。AI BPOは現在においてZero-touch automationを提供する現実的手法であるとも言えます。

AI BPOのよくある勘違い - AI Agent with Human ≠ Human with AI tools
画像
AI Agent with HumanとHuman with AI toolsの違い
AI BPOはAIエージェントの一提供形態です。ここでよくある誤用を整理します。AI Agent with HumanとHuman with AI toolsの違いです。これらは一見似ていますが設計思想が異なります。

AI Agent with Human: AIがワークフロー全体を制御し、必要なタイミングで人を呼び出す
Human with AI tools: 人がワークフロー全体を制御し、必要なタイミングでAIを呼び出す

という違いになります。LayerXではAI BPOを「AI Agent with Human」と定義しています。

AI Agent with Humanとは
例えばエージェントが呼び出すtool callに対して、人の確認を挟まない場合/挟む場合の実装上の違いは僅かです。

// Toolを確認なしに自律実行
const autonomousTool = tool({
  description: "...",
  inputSchema: ...,
  execute: async ({...}) => {...}
});

// Toolの実行を人間が確認
const humanInTheLoopTool = tool({
  description: "...",
  inputSchema: ...,
  needsApproval: true,  // 人の承認が必要
  execute: async ({...}) => {...}
});

copy
needsApproval: true——この1行を加えるだけです。このneedsApprovalの向き先が社外オペレーターになるのがAI BPOです。

とはいえ、これはかなり単純化した例で、実際の実装はもっと複雑です。人がチェックしやすい画面設計、データ修正機能、修正内容を次回に活かすフィードバック機構などが必要です。しかし、考えることの本質はAIエージェント開発とAI BPO開発で同じです。

重要なのはAIエージェントが主体であることです。その上で、人の補完・修正がAIの精度改善につながる機構にします。この機構により、徐々にAIのタスクカバレッジ・精度が上がり、人の負担は減っていきます。この改善サイクルを構造的に組み込むことがAI BPOの核心です。

Human with AI toolsとは
Human with AI toolsなBPOは例えば、こんなオペレーション設計になります。

オペレーターが主体になって専用画面を見ながら作業します。その作業の中でチャットボットで質問をしながら、タスクをこなします。オペレーターの生産性を上げるためにAI toolsを整備していきます。ここでは、エージェントの開発よりも、チャットボットに専門知識であったり、システム知識を教え込むといった知識注入が開発の主体になると思います。

この形では自動化率向上のフィードバックループが構造的に組み込まれにくいです。どこかの段階で人の認知限界にぶつかり、受けられる案件数に限界が出ます。アウトカムも人によってボラティリティが出ます。

こういった取り組みは素晴らしい取り組みだと思いますが、当社定義でのAI BPOではありません。AI BPOの改善とは、人のオペレーション改善ではなく、AIエージェントの改善なのです。

AI BPOの利点
1. AIの進化を待たずに業務の自動化を実現

現在のAIの知性はギザギザです。KarpathyはAIの能力を動物ではなくゴーストに例えています。

ある領域では驚異的な能力を示す一方、別の領域では人がびっくりするようなミスをします。このミスをどう制御するかが、現在のAI活用の鍵です。

AIの進化は急激です。ある時点でのモデルでは不可能だったタスクが、次のモデルでは驚くほどシンプルな指示で解けたりします。LayerX社内でも、様々なコーポレート業務に対してのベンチマークを取っていますが、最上位モデルを使うことで今まで不可能だったタスクがアンロックされるというのを何度も目撃してきました。

しかし、ユーザーは待ってくれません。「AIがいずれ進化するから、それに備えてこのエージェントを入れてください」と言って入れてくれる会社は多くありません。ユーザーは「今この瞬間」完全なサービスを求めています。

我々は「AIが賢くなるのを待つ」というのは悪い戦略だと思っています。「AIが賢くなる」ことを前提に置き、その恩恵を受けるポジショニングを先んじて取ることが良い戦略と考えています。「今Builderとして動く」ことを大事にしています。

AI BPOなら、AIの苦手な部分を人がカバーし、今この瞬間から確実に業務を完遂できます。裏側ではベンチマークを取り続け、業務を通じて溜まる学習データを活用しながら、AIの進化に伴い、自動化の範囲を徐々に広げていけます。「先に提供し、後から置き換える」——これがAI BPOの明確な利点です。

2. デプロイ速度

AIエージェント/AI BPOは「オペレーション"そのもの"をソフトウェア」として扱います。この前提で考えた時、AI BPOの利点は「デプロイ速度」です。

例えば、最上位モデルによってエージェントの能力がアンロックされ、オペレーションの前提が変わる際、AI BPOでは自社オペレーションを変えるだけで全顧客が恩恵を受けられます。AI BPOでない場合は、各社に「オペレーションを変えてください」という説得とオンボーディングが必要です。

エージェントの自律度が現状では100%にはならないので、必ずどこかでオペレーションがはさまります。AI BPOとはこのオペレーションの部分のクラウド化と捉えることもできます。これはSaaSがデスクトップアプリに対して持っていた優位性と似ています。AIの進化スピードが速い現状では、この「デプロイ速度」が強烈な利点になります。

3. 営業のシンプル化

AI BPOは「仕事課金」になります。そしてAI BPOは複数エージェントの「パッケージ」として機能します。

請求書受領代行の裏側では、メール認識Agent、パスコード抽出Agent、ブラウザ操作Agent、テナント割当Agent…と無数のエージェントが動いています。これらに個別に値付けするのは難しいですし、仮に値付けしたとしても、営業の説得コストが高いです。

AI BPOなら「請求書1枚あたりXXX円」で済みます。業務全体のコストとの比較が可能になり、営業もプライシングもシンプルになります。

シンプルなものはスケールします。これは見過ごされがちなAI BPOの強力な利点です。

今後は、債権回収代行、給与計算代行、コスト削減代行…etc、自社のAI native SaaSを基盤としたAI BPOの領域に順次拡大していきます。

究極の目標は「zero-day close」——月次決算が即時に締まっている状態の実現です。AI BPOで月次決算の締めそのものを代行します。月次決算のような複雑で長いタスク（Long-Horizon Task）をAIエージェントで解けるようにし、バックオフィスの長年の夢を実現していきます。

このようにバクラクは「SaaS」から「仕事の完了そのものを提供する」事業へと進化していきます。

バクラクAI BPOでの技術的チャレンジ
LLMを中心においたカスタマイズレイヤー

バクラクAI BPOは、AI native SaaS + AIエージェント + Human in the Loopの組み合わせで構成されています。

受領代行のような業務は、とても標準化しやすいです。一方、承認代行や照合代行では各社の業務特性が異なり、標準化が難しいです。

そこでAI native SaaSの上に、LLMを中心としたカスタマイズレイヤーを作っています。具体的には「Agent/Workflow Studio+ナレッジベース, Memory, Tool Inventory, Eval」を揃えたプラットフォーム層です。これにより、AI BPOを高速で立ち上げつつ、個社の特性も吸収できる基盤を構築しています。

特に、どのようなタスク粒度・確認のUXにすれば人が確認しやすいか?エージェントを評価・改善していくためのEval/Tracebackをどう設計するか?が大きな技術的チャレンジです。

使えば使うほど賢くなるソフトウェア

バクラクAI BPOが目指すのは「使えば使うほど賢くなるソフトウェア」です。この形を作ることがソフトウェア産業のエンドゲームだと考えています。

4. ファインチューニングの逆襲

この5つの予測はAIを実社会に適用し、「使えば使うほど賢くなるソフトウェア」を完成させるためにとても重要な技術の未来予測です。この方向性に技術投資していきます。
