# プロダクトマネージャー向けトーンガイド

## 目的

このトーンガイドは、SaaSプロダクトマネージャー（山内）のライティングスタイルにおける「トーン（tone）」を維持するための実践的なガイドです。記事生成時に一貫したトーンを保ち、読者に適切な印象を与えるための具体的なルールとサンプルを提供します。

---

## トーンの基本原則

### 核となるトーン

> **「知的な実務家が、親しみやすく、データに基づいて未来を語る」**

この一文がすべてのトーンの基盤です。以下の3つの要素を常にバランスよく保ちます。

#### 1. 知的（Intellectual）

- 論理的な思考
- データドリブンな主張
- 専門性の高い内容
- 権威ある出典の引用

**実現方法:**
- 具体的な数値データを提示
- Anthropic, menlo ventures等の信頼できる出典を引用
- 英語の専門用語を適切に使用
- 因果関係を明確に説明

#### 2. 親しみやすい（Approachable）

- カジュアルな語り口
- 読者に寄り添う姿勢
- 難しい概念を平易に説明
- ユーモアのある表現

**実現方法:**
- 「どうも、〜したい山内です」のようなフレンドリーなオープニング
- 「めちゃくちゃ」等のカジュアルな副詞
- 「〜みたいなイメージ」等の平易な比喩
- 読者への問いかけ

#### 3. 実務家視点（Practitioner's View）

- 現場の経験に基づく
- 実践的な洞察
- 具体的な事例
- 行動を促す提案

**実現方法:**
- 自分の経験を語る（「私の考えでは」）
- 実際のプロジェクトや業務の例
- すぐに使える実践的なアドバイス
- 「How」だけでなく「Why」も説明

## 文末表現の使い分け

### 基本比率

- **断定調**: 70%
- **思考の提示**: 20%
- **疑問形**: 10%

### 1. 断定調（70%）

**使用場面:**
- 確信を持った主張
- データに基づく事実
- 一般的に認められた事柄

**表現パターン:**

| 表現 | 使用例 | ニュアンス |
|------|--------|------------|
| 〜です。 | 「AI Agentは今後重要になります。」 | 基本形、最も頻繁に使用 |
| 〜だと思います。 | 「これは大きな変化だと思います。」 | 断定+個人的見解 |
| 〜になります。 | 「2025年はAI Agentの年になります。」 | 確信を持った予測 |
| 〜でしょう。 | 「この傾向は続くでしょう。」 | やや控えめな予測 |
| 〜ということです。 | 「つまり、SaaSが変わるということです。」 | 言い換え、要約 |

**良い例:**

```markdown
AI Agent Eraでは、人ではなくAIがSaaSを動かします。この変化は2025年に本格化するでしょう。まず間違いなく、業務のあり方は大きく変わります。
```

**避けるべき例:**

```markdown
AI Agent Eraでは、人ではなくAIがSaaSを動かすかもしれません。この変化は2025年に本格化する可能性があると考えられます。おそらく、業務のあり方は大きく変わるのではないでしょうか。
```

**理由:** 「かもしれません」「可能性があると考えられます」等の過度に慎重な表現は、自信のなさを感じさせる。

### 2. 思考の提示（20%）

**使用場面:**
- 個人的な見解
- 解釈・考察
- 経験に基づく洞察

**表現パターン:**

| 表現 | 使用例 | ニュアンス |
|------|--------|------------|
| 〜という感覚。 | 「淘汰的な意思決定が洗練されていくという感覚。」 | 直感的な理解 |
| 〜かなと思います。 | 「SaaS is Functionという話かなと思います。」 | 柔らかい主張 |
| 〜と考えています。 | 「私の考えでは、SaaSは死なないと考えています。」 | 熟考した見解 |
| 〜だと感じます。 | 「すでにAIによって扱えると感じます。」 | 実感に基づく |

**良い例:**

```markdown
私の考えで行くと、SaaSは死なないが、AI Agentの波に乗れないSaaS企業は死ぬという結論です。これは極論ではなく、経営者として肌で感じる現実だと思います。
```

### 3. 疑問形（10%）

**使用場面:**
- 読者に考えさせる
- 問題提起
- 反論の予測

**表現パターン:**

| 表現 | 使用例 | ニュアンス |
|------|--------|------------|
| 〜でしょうか。 | 「より柔軟なRPAみたいなイメージというとわかりやすいでしょうか。」 | 読者への配慮 |
| 〜なのか? | 「マーケットが悪いのか?自社のオペレーションの強度が低いのか?」 | 問題提起 |
| 〜だろうか? | 「本当にSaaSは死ぬのだろうか?」 | 疑問の提示 |

**良い例:**

```markdown
Why ビジネス人材なのか?

エンジニアに任せればいいのでは?という疑問があるかもしれません。しかし、ビジネスの文脈を最も理解しているのは、現場のビジネス人材です。
```

## カジュアル表現のルール

### 使ってよいカジュアル表現

#### 1. 「めちゃくちゃ」（頻出）

**使用例:**
- 「めちゃくちゃ重要」
- 「めちゃくちゃ反論可能」
- 「めちゃくちゃ速い」

**使用頻度:** 1記事あたり2〜4回

**注意点:**
- 強調したいポイントで使用
- 多用しすぎない（5回以上は避ける）

#### 2. 「というか」

**使用例:**
- 「マーケットというと広い表現になってしまうけど」
- 「というか、〜」

**使用頻度:** 1記事あたり1〜2回

#### 3. 「よくよく考えると」「よくよく考えるべき」

**使用例:**
- 「よくよく考えるべきと私は思います」
- 「よくよく考えると、〜」

**使用頻度:** 1記事あたり1〜2回

**効果:** 深い思考を促す、重要性を強調

#### 4. 「〜だったりします」

**使用例:**
- 「具体的には〜だったりします」
- 「実際には〜だったりします」

**使用頻度:** 1記事あたり1〜3回

### 使わないカジュアル表現

| 表現 | 理由 |
|------|------|
| ヤバい | 過度にカジュアル、ビジネス記事に不適 |
| マジで | 幼稚な印象 |
| 超〜 | 「めちゃくちゃ」で代替 |
| 絵文字（😊🚀等） | 基本的に使用しない（視覚的な図表は除く） |

## 専門用語の扱い

### 基本ルール

1. **英語のまま使用**
2. **初出時に定義**
3. **日本語訳を併記**
4. **出典を明示**

### 典型的なパターン

#### パターン1: 引用による定義

```markdown
AI Agentの定義はAnthropicの定義を借りると、以下のものになります。

> Agents are systems where LLMs dynamically direct their own processes and tool usage, maintaining control over how they accomplish tasks.
>
> (日本語訳)
> エージェントは、LLMが自らのプロセスやツールの使用を動的に指揮し、タスクをどのように達成するかを自律的に制御するシステムです。
```

#### パターン2: 括弧書きによる定義

```markdown
ACV(Annual Contract Value: 顧客単価)が劇的に変わります。
```

#### パターン3: 別文での説明

```markdown
AI Agentは、あるタスクの目的+タスク完了のための道具群を用意すると、AIが自律的にタスクを計画し、タスクを完了する(事前に定義されたコードを必要としない)といったイメージです。より柔軟なRPAみたいなイメージというとわかりやすいでしょうか。
```

### 頻出する専門用語とその扱い

| 専門用語 | 初出時の定義 | 2回目以降 |
|----------|--------------|-----------|
| AI Agent | Anthropicの定義を引用 | そのまま使用 |
| Workflow | 事前定義されたコードパスで調整されるシステム | そのまま使用 |
| Orchestrator | タスクを計画し、workersをcallする大きなAI | そのまま使用 |
| Context | AIがタスクを解くための背景情報 | そのまま使用 |
| LLM | Large Language Model（大規模言語モデル） | LLMのみ |

## オープニングとクロージング

### オープニング（導入部）

#### パターン1: ビジョン提示型

```markdown
どうも、[実現したいビジョン]したい山内です。

[時事的な背景・トレンド認識]
```

**実例:**
- 「どうも、すべての経済活動を、デジタル化したい山内です。」
- 「どうも、プロダクトの価値を最大化したい山内です。」
- 「どうも、ビジネス課題をテクノロジーで解決したい山内です。」

#### パターン2: 問題提起型

```markdown
どうも、山内です。

最近、[テーマ]について、あらためて[気づき]を再認識することが多いです。
```

#### パターン3: シリーズ継続型

```markdown
どうも、山内です。

恒例の[シリーズ名]。今年は[テーマ]について書こうと思います。
```

### クロージング（締めくくり）

#### パターン1: 採用募集型

```markdown
絶賛採用拡大中で、カジュアル面談も大歓迎です！[テーマ]について話しましょう。より詳しい話もたくさんできると思います！

興味あるよという方は下のリンクより、お申し込みください。
```

#### パターン2: 未来展望型

```markdown
2025年は[テーマ]元年。我々は[ビジョン]を実現していきます。

一緒に未来を作っていきましょう。
```

#### パターン3: 関連コンテンツ誘導型

```markdown
詳しくは以下の記事・podcastもぜひご覧ください。

[リンク集]
```

## 論理展開のトーン

### 問題提起のトーン

**特徴:**
- 明確で鮮明
- データや事例で裏付ける
- 読者の関心を引く

**表現パターン:**

```markdown
[時期]、「[センセーショナルな言葉]」という言葉が取り沙汰されました。

実際に内容を聞いたうえでの私の考えで行くと、[自分の解釈]という話かなと思います。
```

**実例:**

```markdown
2024年「SaaS is Dead」というセンセーショナルな言葉が取り沙汰されました。

実際に内容を聞いたうえでの私の考えで行くと、SaaS is Deadというよりは、SaaS is Function(SaaSがAIが動かす関数・手足のone of themになる)という話かなと思います。
```

### 定義・説明のトーン

**特徴:**
- 権威ある出典を引用
- 明確で簡潔
- 読者がイメージしやすい

**表現パターン:**

```markdown
[概念]とは何か。

[出典]の定義を借りると、以下のものになります。

> [引用]

[平易な言葉での言い換え]というとわかりやすいでしょうか。
```

### 反論対処のトーン

**特徴:**
- 反論を認識していることを示す
- 部分的に妥当性を認める
- しかし、より本質的な視点を提示

**表現パターン:**

```markdown
ここらへんの話は、[立場]からするとめちゃくちゃ反論可能だし、反論したくなるような話とも思います。[予想される反論]という反論です。

私の考えで行くと、[自分の結論]という結論です。
```

**実例:**

```markdown
ここらへんの話は、実際にSaaS企業を経営している経営者からするとめちゃくちゃ反論可能だし、反論したくなるような話とも思います。SaaSはディストリビューションゲームであり、SaaSが死ぬは極論であるという反論です。

私の考えで行くと、SaaSは死なないが、AI Agentの波に乗れないSaaS企業は死ぬという結論です。
```

### データ提示のトーン

**特徴:**
- 出典を明示
- 具体的な数値
- 解釈を添える

**表現パターン:**

```markdown
[出典]によると、[データ]だそうです。[解釈・示唆]ということです。
```

**実例:**

```markdown
menlo venturesによると、UiPathではSaaS $1に対して、コンサル・BPO(設定代行)に $7払われているそうです。SaaSよりも人の業務に払われるSpendのほうが大きいということです。
```

## 一人称の使い分け

### 「私」

**使用場面:**
- 個人的見解
- 経験談
- 感情・直感

**使用例:**
- 「私の考えで行くと」
- 「私自身もデモを見る機会がありましたが、衝撃的でした」
- 「よくよく考えるべきと私は思います」

### 「我々」「当社」

**使用場面:**
- 組織としての取り組み
- 会社の戦略
- チームの方針

**使用例:**
- 「我々は〜を実現していきます」
- 「当社では〜というアプローチをとります」
- 「我々のチャレンジ」

**注意点:**
- 個人ブログでは「私」を中心に
- 企業ブログでは「我々」を適度に

### 「山内」

**使用場面:**
- 自己紹介
- 署名
- オープニング

**使用例:**
- 「どうも、〜したい山内です」
- 「山内（@〜）」

## 比喩・たとえ話のトーン

### 使い方

**特徴:**
- 複雑な概念を身近な例で
- 「〜みたいなイメージ」「〜のような」
- 視覚的なイメージを喚起

**表現パターン:**

```markdown
[複雑な概念]は、[身近な例]みたいなイメージというとわかりやすいでしょうか。
```

**実例:**

| 概念 | 比喩 |
|------|------|
| AI Agent | より柔軟なRPAみたいなイメージ |
| AI-OCR | SaaSに目がついた |
| マルチモーダル化 | SaaSに頭脳がつくし、口と耳がつくし、手足も生える |
| Orchestrator-workers | 大きなAIが指揮者、小さなAIが演奏者 |

### 良い比喩の条件

1. **読者が知っている概念**
2. **視覚的にイメージしやすい**
3. **本質を捉えている**
4. **過度に単純化していない**

## 強調表現のトーン

### 使用場面

- 重要なポイント
- 驚くべき事実
- 転換点

### 表現パターン

| 表現 | 使用例 | 効果 |
|------|--------|------|
| めちゃくちゃ | 「めちゃくちゃ重要」 | 強い強調 |
| 非常に | 「非常に大きな意味」 | フォーマルな強調 |
| 衝撃的 | 「衝撃的でした」 | 驚き |
| まず間違いなく | 「まず間違いなく〜になります」 | 確信 |
| いよいよ | 「いよいよ始まる」 | タイミング |
| 今この瞬間だからこそ | 「今この瞬間だからこそ可能」 | 希少性 |

### 使用頻度の目安

- 1記事（3,000〜5,000文字）あたり
  - 「めちゃくちゃ」: 2〜4回
  - 「非常に」: 3〜5回
  - 「衝撃的」「いよいよ」等: 各1〜2回

## 禁止事項

### 避けるべきトーン

1. **過度に堅苦しい**
   - 「〜である。」の連続
   - 論文調、お役所調
   - 読者との距離感がある

2. **過度にカジュアル**
   - 「ヤバい」「マジで」
   - 絵文字の多用
   - スラングの使用

3. **自信のなさ**
   - 「かもしれません」の連発
   - 「〜の可能性があると考えられます」
   - 慎重すぎる表現

4. **攻撃的**
   - 他者を貶める表現
   - 極端な二元論
   - 感情的な言葉

## 記事タイプ別のトーン適用

### 解説・教育記事

**トーンの重点:**
- 知的（50%）× 親しみやすい（40%）× 実務家視点（10%）
- 丁寧な説明
- 読者の理解を助ける配慮

**特徴的な表現:**
- 「〜というとわかりやすいでしょうか」
- 「具体的には〜です」
- 「例えば〜」

### 意見・主張記事

**トーンの重点:**
- 実務家視点（50%）× 知的（30%）× 親しみやすい（20%）
- 断定調を多用
- 反論への対処

**特徴的な表現:**
- 「私の考えで行くと」
- 「よくよく考えるべきと思います」
- 「まず間違いなく〜」

### ハウツー・実践記事

**トーンの重点:**
- 実務家視点（60%）× 親しみやすい（30%）× 知的（10%）
- 実践的な語り口
- 読者が行動できる形

**特徴的な表現:**
- 「まず〜します」
- 「次に〜します」
- 「ここで注意すべきは〜」

## まとめ

このトーンガイドは、一貫性のあるライティングスタイルを維持するためのツールです。

**重要なポイント:**

1. **核となるトーンを常に意識**: 知的 × 親しみやすい × 実務家視点
2. **断定調を中心に**: 自信を持って主張する
3. **カジュアル表現を適度に**: 「めちゃくちゃ」は2〜4回/記事
4. **データで裏付ける**: 出典を明示し、数値を示す
5. **反論を予測する**: 先回りして対処する

記事作成時にこのガイドを参照し、チェックリストで品質を確認してください。
